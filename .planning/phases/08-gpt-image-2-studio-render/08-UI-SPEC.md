## Component Inventory

These are the specific UI elements this phase adds or modifies. All must follow existing studio.css patterns exactly.

### 1. Studio Render Button ‚Äî HUD top-right

**Current state (stub):** Already exists in `stage.jsx` hud-tr seg as:
```
h('button', { className: 'on', onClick: () => { setMode('render'); startRender(); } },
  h(I.Sparkle, { size: 13 }), 'Studio Render')
```

**Contract:**
- Idle state: `.seg button` with `I.Sparkle` icon + label `"Studio Render"` ‚Äî same as current
- During render: button disabled (no `onClick`), opacity 0.5, cursor `not-allowed`
- After render complete: label changes to `"Re-render"` ‚Äî same `.seg button` styling, not `.on` class
- Button placement: hud-tr, second `.seg` row, after the background toggle ‚Äî no layout change

### 2. Render Progress Veil ‚Äî existing `.render-veil`

**Current stub copy (replace):**
- Heading: `"Compositing your wrap"` ‚Üí replace with `"Rendering studio shot‚Ä¶"`
- Body: `"Re-lighting the paint, reflections and material depth."` ‚Üí replace with `"Compositing into the M&C studio bay ‚Äî usually 15‚Äì30 seconds."`
- Progress percentage display: `Math.round(renderPct) + '% ¬∑ ~12s'` ‚Üí `Math.round(renderPct) + '%'` ‚Äî drop the hard time promise

**Contract:**
- `.render-card .rk` eyebrow: `"Studio Render"` (unchanged)
- `.render-card h3`: `"Rendering studio shot‚Ä¶"` (updated)
- `.render-card p`: `"Compositing into the M&C studio bay ‚Äî usually 15‚Äì30 seconds."` (updated)
- Progress bar: `.render-bar` + `.render-bar i` ‚Äî same CSS, `width: renderPct + '%'` ‚Äî no change
- Percentage label: `Math.round(renderPct) + '%'` ‚Äî no time estimate appended
- No spinner ‚Äî use the existing sweeping bar only
- Veil backdrop: `rgba(6,7,9,.72)` with `backdrop-filter: blur(3px)` ‚Äî DO NOT change

### 3. Before/After Slider ‚Äî `renderUrl` state

**Contract (from CONTEXT.md):**
- When `renderUrl` is null: before = original photo (`originalUrl`), after = CSS recolour preview (`displayUrl`) ‚Äî existing behaviour unchanged
- When `renderUrl` is set: before = original photo (`originalUrl`), after = studio render (`renderUrl`)
- BA tag labels when `renderUrl` is set: "Studio Render" (after, left of divider) / "Original" (before, right of divider)
- BA tag labels when `renderUrl` is null: "Wrapped" / "No wrap" ‚Äî existing strings unchanged
- BA tags: `.ba-tag.after` and `.ba-tag.before` ‚Äî no new CSS classes needed

### 4. Toast Notifications

Two distinct messages (from CONTEXT.md). Use existing `flash(msg)` utility ‚Äî no new toast variants.

| Trigger | Toast copy |
|---------|------------|
| Render success | `"Studio render ready"` |
| Rate-limit / quota exceeded | `"Too many renders ‚Äî try again shortly"` |
| API / network error | `"Render failed ‚Äî try again"` |
| No car uploaded when render triggered | `"Upload your car photo first"` |

Toast anatomy: existing `.toast` with `I.Check` icon ‚Äî same for all four cases. Do not add a red/error variant; the copy distinction is sufficient.

### 5. Canvas Export Helper ‚Äî `window.__wrapRenderCanvas`

Follows the `window.__wrapDownload` pattern already in `stage.jsx`.

**Contract:**
- Registered in `stage.jsx` via `useEffect` on `[displayUrl]`
- Function name: `window.__wrapRenderCanvas`
- Returns: `Promise<Blob>` ‚Äî PNG blob of the current stage canvas at native resolution
- Error case: returns `null` (not a rejected promise) ‚Äî caller checks for null before fetch

| BA slider | render ready | Compares `originalUrl` (right) vs `renderUrl` (left) |
| BA tags | render ready | "Studio Render" (left) / "Original" (right) |
