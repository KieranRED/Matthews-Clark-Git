## Component Inventory

### 1. Subscribe Settings Row

A standard `.set-row` entry appended to the INTEGRATIONS section in `screens-settings.jsx`. No new CSS classes required.

**Visibility gate:** Rendered only when `window.matchMedia('(display-mode: standalone)').matches` is true. Server-side: always render as `null`; gate is client-side only (useEffect + useState).

**States and their visual representation:**

| State | Icon bg | Left border | Row name | Meta text | Chevron |
|-------|---------|-------------|----------|-----------|---------|
| `checking` (initial) | `rgba(255,255,255,.04)` | none | "Notifications" | "CHECKING…" (fg-3) | shown |
| `unsupported` | `rgba(255,255,255,.04)` | none | "Notifications" | "NOT SUPPORTED ON THIS DEVICE" (fg-3) | hidden |
| `unsubscribed` | `rgba(255,255,255,.04)` | none | "Enable notifications" | "TAP TO SUBSCRIBE · PWA ONLY" (fg-3) | shown |
| `requesting` | `rgba(31,79,255,.12)` | none | "Subscribing…" | "REQUESTING PERMISSION" (fg-3) | hidden |
| `subscribed` | `rgba(31,79,255,.12)` | `3px solid #1F4FFF` | "Notifications" | `● ACTIVE` (success dot `#27AE60` + fg-3 text) | shown |
| `error` | `rgba(235,87,87,.12)` | none | "Notifications" | error message string (fg-3, uppercase, truncated) | shown |

The icon used in all states: `Icon.send` (matches existing Telegram row convention).

**No unsubscribe action in this phase.** Tapping the row when `subscribed` does nothing (or shows a no-op — do not wire an onClick, leave `cursor: default`, `opacity: 0.9`).

### 2. Service Worker Registration

Invisible to the user. Registered via `useEffect` in the CRM layout client component. No UI element.

### 3. System Push Notification

Delivered by the OS (iOS/Android). No custom styling. Payload drives system rendering:
- Title: `"New WhatsApp"`
- Body: `"{leadName}: {messagePreview}"`

No in-app notification UI needed for this phase.

## Interaction Contract

### Subscribe flow (happy path)

1. User opens CRM in PWA standalone mode → row appears in INTEGRATIONS section, state `checking` → transitions to `unsubscribed` within 200ms.
2. User taps row → state transitions to `requesting` (no onClick on subscribed state).
3. Browser prompts native permission dialog → user grants → state transitions to `subscribed`.
4. `POST /api/push/subscribe` called with PushSubscription JSON → success response → row remains `subscribed`.

### Error path

- Permission denied by user → state `error`, meta shows "PERMISSION DENIED BY USER".
- `POST /api/push/subscribe` fails → state `error`, meta shows truncated error message.
- Row remains tappable in `error` state to allow retry.

### Transition timing

All state transitions: instant (no animation delay). The `.screen` CSS already provides a 240ms fadeIn on mount — do not add additional transitions to the row itself.

## Standalone Gate Implementation

```jsx
// Mount check — renders null outside standalone PWA
const [isStandalone, setIsStandalone] = useState(false);
useEffect(() => {
  setIsStandalone(window.matchMedia('(display-mode: standalone)').matches);
}, []);
if (!isStandalone) return null;
```

This must wrap the subscribe row at the call-site in `screens-settings.jsx`, not inside the row component itself, so that the INTEGRATIONS section layout is not affected when the row is absent.
