# Phase 5 — UI Design Contract: Integration & Catalogue

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope note:** The design system prototype (studio.css + JSX files) is already complete. This contract documents its existing visual language so future phases stay consistent, then specifies the delta decisions for the real catalogue: swatch grid density, metallic finish chip, quote modal redesign, low-confidence hex treatment, and TweaksPanel suppression.

## Focal Point Hierarchy

Primary focal point: the swatch grid (`.cat-scroll`). The stage area is secondary. The panel is tertiary.

All layout decisions — panel width, stage proportions, scrollable grid — reinforce this hierarchy. The swatch grid must receive the most visual real estate and the clearest contrast treatment.

## Typography

All sizes from studio.css. Executor must not change these values.

**Four declared sizes, two weights.** No fifth size or third weight may be introduced.

| Role | Size | Weight | Line Height | Font | Usage |
|------|------|--------|-------------|------|-------|
| Body | 15px | 400 (regular) | 1.5 | Inter Tight | All prose, form inputs, detail card text |
| Label / mono | 10–11px | 400 (regular) | 1.4 | JetBrains Mono | Product codes, finish chips, UI captions — `letter-spacing: .14–.22em; text-transform: uppercase` |
| Heading | 16px | 800 (extrabold) | 1.15 | Archivo | Panel title, detail card name, modal h3 |
| Display | 21px | 400 (regular Anton) | 1 | Anton | Topbar wordmark "MATTHEWS&CLARK" — `letter-spacing: .10em; text-transform: uppercase` |

**Swatch card micro-variants (contextual variants of the mono/label tier — not distinct sizes):**

`.sw-name` and `.sw-code` are both contextual sub-variants of the label/mono tier rendered inside `.sw-meta`. They do not constitute additional declared sizes.

- `.sw-name`: 12px, weight 600, line-height 1.25, 2-line clamp — treated as the upper bound of the mono tier, using the second weight (600)
- `.sw-code`: 9px, weight 400, JetBrains Mono, color `rgba(255,255,255,.42)` — treated as the lower bound of the mono tier, used only in this one dense-chip context

Both render in JetBrains Mono and fall within the label/mono tier's purpose (product identifiers in constrained chip real estate). Executor must not apply these sizes outside `.sw-meta`.

**Do not introduce a fifth font size or a third weight.** The two weights in use are 400 and 600/700/800 (Archivo uses 700–800 for headings; treat as "heavy" = the second weight tier).

## Color

Dark cinematic palette. All values from studio.css `:root`. Executor must reference CSS variables — do not hardcode hex.

| Role | Variable | Value | Usage |
|------|----------|-------|-------|
| Dominant (60%) | `--ink` / `--bay` | `#0A0A0A` / `#0d0e11` | Full-page background (`--ink`), stage area (`--bay`) |
| Secondary (30%) | panel bg | `#0c0c0e` | `.panel` background; `.q-summary` `#0b0b0d`; input fields `#141416` |
| Accent (10%) | `--accent` | `#1F4FFF` | See reserved list below |
| Accent bright | `--accent-bright` | `#4A78FF` | Hover state of primary buttons only |
| Accent deep | `--accent-deep` | `#1638CC` | Active/pressed state of primary buttons only |
| Separator | `--line` | `rgba(255,255,255,.08)` | All panel dividers, card borders |
| Separator stronger | `--line-2` | `rgba(255,255,255,.14)` | Input borders, button borders, tier badge borders |
| Destructive | n/a | not defined in prototype | No destructive actions in Phase 5 |

**Accent (`#1F4FFF`) is reserved exclusively for:**
1. Selected swatch border + double ring: `.swatch.on { border-color: var(--accent); box-shadow: 0 0 0 1px var(--accent) }`
2. Active brand tab underline: `.brand-tabs button.on { border-bottom-color: var(--accent) }`
3. Active finish chip fill: `.fchip.on { background: #fff; color: #0a0a0a }` — note: the active finish chip intentionally uses inverted white/black, NOT accent blue
4. Primary CTA button background: `.btn--primary { background: var(--accent) }`
5. Before/after divider bar: `.ba-divider { background: var(--accent) }`
6. B/A "after" tag: `.ba-tag.after { background: rgba(31,79,255,.85) }`
7. Detail card pro-tip border + background tint: `rgba(31,79,255,.08)` / `rgba(31,79,255,.28)`
8. Stage drop veil dashed border
9. Specialist tier text on `.sw-tier.specialist`
10. Search input focus ring

Accent must NOT be used on: general hover states, body text, secondary buttons, or any informational labels.

## Catalogue: Real Data Decisions

### Grid density for 375 items

The demo catalogue had ~110 items. With 375 items:

- **Grid column width:** `minmax(110px, 1fr)` — inherited from prototype. Do NOT increase to wider columns. At 416px panel width this yields ~3 columns in default placement, giving adequate density without swatch images looking cramped.
- **Swatch chip height:** 74px — unchanged. Real swatch PNGs at this height are readable.
- **Scrollbar:** styled custom scrollbar from studio.css (width: 9px, thumb `rgba(255,255,255,.12)`) — retain as-is.
- **Count display:** the `{filtered} / {total}` counter in `.panel-title .ct` is load-bearing UX for a 375-item list — keep it visible at all times.
- **Virtual scrolling:** Phase 5 does NOT require virtual scrolling. 375 items at ~100px height each = ~37,500px total — modern browsers handle this without jank. Defer if profiling shows a problem.

### New "Metallic" finish chip

Add `{ key: 'metallic', label: 'Metallic' }` as the 4th entry in `window.FINISHES` (after Matte, before Chrome). This maps to items with `finish: 'metallic'` from the real data (do NOT remap metallic → satin; keep it as a first-class key so users can filter it).

Finish chip visual contract — the `metallic` chip follows the same `.fchip` styling as all others. No special icon or colour differentiation in Phase 5.

**Updated FINISHES order:**
1. Gloss
2. Satin
3. Matte
4. Metallic ← new
5. Chrome
6. Colour-shift
7. Carbon ← add (real data has `carbon` finish)
8. PPF Clear
9. PPF Matte

Note: `brushed` finish from real data maps to satin (CONTEXT.md decision). `ppf-colour` maps to ppf-clear or ppf-matte based on slug. Both transformations happen in `build-catalogue.js` — no new finish chips needed for these.

### Low hex_confidence treatment

The real `wrap-colours.json` has `hex_confidence: "low"` on certain entries (primarily chrome and colour-shift films where the swatch is a gradient/angle-dependent colour). These are colour approximations only.

**Swatch chip for low-confidence entries:**

When `hex_confidence === 'low'`, render an additional small indicator on the swatch chip:

- Position: bottom-left of `.sw-chip`, same z-layer as `.sw-tier`
- Element: `<span class="sw-approx">~</span>` (tilde symbol — universal "approximate" signal)
- Style: same pill treatment as `.sw-tier` — `font-family: var(--f-mono); font-size: 8px; letter-spacing: .1em; background: rgba(10,10,10,.6); border: 1px solid var(--line-2); color: rgba(255,255,255,.55); padding: 2px 5px`
- Tooltip (title attr): `"Colour is approximate — {finish} films shift with angle and light"`

**Detail card for low-confidence entries:**

In the `.d-specs` grid, add a fourth spec cell (after Warranty):

- Key: "COLOUR"
- Value: "Approx."
- Style: same as existing `.d-spec` cells — no special colour treatment

**Do NOT hide low-confidence items or render them differently in the grid.** The `~` indicator is the only treatment; it informs without stigmatising.

### Swatch images from Vercel Blob

Each catalogue entry will have `swatchUrl` populated by `build-catalogue.js`. The swatch chip renders:

```
<img src={sw.swatchUrl} alt={sw.name} style={{ width: '100%', height: '100%', objectFit: 'cover', display: 'block' }} />
```

placed as the FIRST child of `.sw-chip`, before any overlay spans (gloss/matte/chrome). The `chipBg(sw)` hex fallback remains as the `background` on `.sw-chip` — it shows while the image loads.

If `swatchUrl` is null/undefined (e.g. during local dev before build script runs), fall back to hex-only rendering — no broken image icons.

## Quote Modal Redesign

The prototype's `.q-body` is a two-column grid: `.q-summary` (left, colour selection) + `.q-form` (right, form fields).

**Phase 5 changes:**

1. **Remove `.quote-tier` and tier note from the quote footer** (the panel-bottom CTA area). Replace with just the selected colour name + swatch dot. The "Get a quote for this wrap" button remains.

2. **Quote modal `.q-summary` column:** Show the selected colour(s) with `.q-line` items (swatch dot, panel label, colour name, product code). The `.qs-tier` badge in `.q-sum-head` is removed entirely. The `.q-note` text changes to: "We'll confirm availability and come back with a fixed price — usually same day."

3. **Quote modal `.q-form` column:** Four fields only:
   - Name (text input, required)
   - Car: make/model/year (text input, required)
   - WhatsApp/phone (tel input, required)
   - Notes (textarea, optional, 3 rows)
   - Fine print line: "We'll be in touch within one business day. No obligation."
   - Submit button: "Send quote request" (`.btn--primary`, full width)

4. **No price, no tier, no estimate** anywhere in the modal or quote footer. The customer sees colour selection + a clean contact form. M&C quotes fixed price by return.

5. **Submission behaviour (Phase 5):** The form submits to `/api/lead` (existing endpoint). Wire-up of colour/panel data to the lead store is Phase 8 — in Phase 5 the submit button fires the existing API with name/car/whatsapp/notes only and shows the confirmation toast.

## TweaksPanel: Production Suppression

The `TweaksPanel` is a developer-facing edit-mode overlay that communicates with the omelette host via `window.postMessage`. It must be hidden in production.

**Contract:**

- `tweaks-panel.jsx` IS copied to `public/wrap-studio/` and the `<script>` tag IS included in `page.jsx` (the `useTweaks` hook is used by `app.jsx` for `t.placement`, `t.accent`, `t.defaultFinish`, `t.panelMode`, `t.renderSeconds` — removing the file would break those reads).
- `TweaksPanel` renders only when it receives a `__activate_edit_mode` postMessage. In production (Next.js page loaded directly by a customer), no parent frame sends this message — the panel never renders.
- No additional code change is needed to suppress it. It is self-suppressing by design.
- **Do NOT** render `<TweaksPanel>` conditionally on an env var or remove it from the JSX tree. It stays exactly as the prototype has it.
